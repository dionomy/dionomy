# 구현 정책

## 제품 정의

Dionomy는 성인 취미 학원을 위한 모듈형 운영 앱 세팅 서비스다.

원장이 직접 앱을 만드는 노코드 빌더가 아니라, 우리 측 관리자가 학원별 기능, 워딩, 정책, 화면 구성을 세팅하고 원장·강사·수강생은 각자 운영 기능만 사용하는 구조다.

## 기본 아키텍처

초기 구현은 하나의 공통 시스템을 학원별 설정값으로 다르게 보여주는 방식으로 한다.

```txt
공통 백엔드
공통 웹앱
학원별 설정값
학원별 모바일 화면
PWA 또는 WebView 앱
```

초기부터 학원별 네이티브 앱을 따로 개발하지 않는다.

## 사용자 역할

### 우리 측 관리자

관리자는 학원별 앱 세팅을 전담한다.

- 학원 생성
- 학원 유형 선택
- 앱 이름 설정
- 앱 아이콘/스플래시 관리
- 기능 모듈 on/off
- 메뉴명 변경
- 기능별 워딩 변경
- 화면 템플릿 적용
- 예약/보강/회차권 정책 설정
- 원장 계정 발급
- 앱 출시 정보 관리

### 원장

원장은 운영 기능만 사용한다.

- 수강생 등록
- 수업 일정 등록
- 예약 승인/거절
- 출석 처리
- 회차권 부여
- 보강 처리
- 수업 기록 작성
- 공지 작성
- 수강생 메모 관리

원장은 기능 추가, 기능 제거, 화면 구조 변경, 워딩 변경, 정책 구조 변경, 앱 출시 설정 변경을 하지 않는다.

### 강사

강사는 수업 운영에 필요한 기능만 사용한다.

- 담당 수업 확인
- 출석 처리
- 수업 기록 작성
- 수강생 메모 확인

### 수강생

수강생은 모바일 웹/PWA/WebView 앱에서 본인 이용 기능만 사용한다.

- 예약 확인
- 일정 변경 요청
- 잔여 회차 확인
- 보강권 확인
- 수업 기록 확인
- 공지 확인
- 알림 상세 확인

## 모듈 정책

기능은 완전 자유형 빌더가 아니라 정해진 운영 모듈을 학원별로 활성화하는 방식으로 구현한다.

초기 핵심 모듈:

- 예약 관리
- 회차권 관리
- 보강 관리
- 출석 관리
- 수업 기록
- 공지
- 강사 관리
- 공간/룸 관리

확장 후보:

- 원데이 클래스
- 그룹 수업
- 1:1 레슨
- 대기 신청
- 수강생 CRM
- 이탈 감지 AI
- 알림 자동화

## 설정 정책

학원별 차이는 코드 분기가 아니라 설정값으로 처리한다.

설정 대상:

- 기능 on/off
- 메뉴명
- 기능별 워딩
- 화면 템플릿
- 운영 정책
- 브랜딩 정보

워딩 예시:

```txt
공통 기능: LessonPass

바이올린 학원: 회차권, 남은 레슨, 보강권
발레 학원: 수강권, 출석권, 대체 수업
도예 공방: 이용권, 클래스권, 예약 가능 횟수
PT 스튜디오: 세션권, 잔여 세션, 보충 세션
```

내부 도메인 모델은 공통으로 유지하고, 사용자 표시 문구만 학원별 설정으로 바꾼다.

## 운영 정책

기능 on/off보다 학원별 운영 정책 설정을 중요하게 다룬다.

초기 정책 항목:

- 예약 취소 가능 시간
- 일정 변경 가능 시간
- 보강권 유효 기간
- 회차권 만료 기간
- 당일 취소 시 회차 차감 여부
- 원장 승인 필요 여부
- 수강생 직접 예약 가능 여부
- 수업 전 알림 시간
- 예약 가능 인원
- 대기 신청 허용 여부

정책은 하드코딩하지 않고 설정 DB에서 관리한다.

## 화면 정책

화면은 원장이 직접 편집하지 않는다.

화면 구성은 다음 조합으로 만든다.

```txt
템플릿 선택 + 모듈 배치 + 워딩 적용 + 정책 반영
```

초기 템플릿:

- 1:1 레슨형
- 그룹 수업형
- 원데이 클래스형

수강생 모바일 웹의 기본 탭 예시:

```txt
1:1 레슨형: 홈, 내 예약, 내 레슨, 알림, 더보기
그룹 수업형: 홈, 시간표, 내 수업, 알림, 더보기
원데이 클래스형: 홈, 클래스, 내 신청, 알림, 더보기
```

## 앱 제공 정책

초기 제품은 모바일 웹/PWA로 검증한다.

WebView 앱은 유료 고객 또는 출시 필요성이 확인된 학원에 붙인다.

WebView 앱이 담당하는 것:

- WebView 로딩
- 학원 식별값 전달
- 푸시 알림 토큰 등록
- 딥링크 처리
- 파일 업로드/다운로드 보조
- 앱 버전 관리
- 네트워크 상태 처리
- 앱 아이콘/스플래시 적용

WebView 앱에 넣지 않는 것:

- 예약 로직
- 회차권 로직
- 보강 로직
- 수업 기록 로직
- 화면 구성 로직
- 학원별 정책 로직
- 워딩 변경 로직

비즈니스 로직은 공통 웹앱과 백엔드에서 처리한다.

## 데이터 정책

데이터는 운영 데이터와 설정 데이터를 분리해서 생각한다.

주요 저장소:

- 운영 DB: 수강생, 수업, 예약, 출석, 회차권, 보강, 기록
- 설정 DB: 학원, 기능 on/off, 워딩, 정책, 화면 템플릿, 브랜딩
- 캐시: Redis
- 파일 저장소: S3/CloudFront 또는 호환 스토리지

모든 주요 데이터는 학원 단위로 격리 가능한 멀티테넌트 구조를 전제로 한다.

## 백엔드 정책

백엔드는 API Gateway, 인증/권한, 멀티테넌트, 설정 관리, 운영 모듈을 중심으로 설계한다.

주요 책임:

- 인증/권한 관리
- 학원별 테넌트 식별
- 학원별 설정 조회
- 운영 정책 적용
- 예약/회차권/보강/출석/수업기록 처리
- 알림 이벤트 생성
- CRM/AI 분석용 데이터 제공

기술 스택 후보:

- Kotlin + Spring Boot
- PostgreSQL
- Redis

MVP 백엔드는 Kotlin Spring Boot 기반 모듈러 모놀리스로 시작한다. 패키지는 feature-based로 나누고, feature 내부에는 DDD 원칙을 적용한다.

Python FastAPI는 MVP 백엔드가 아니라 v2 AI/LLM worker 후보로 둔다.

## 프론트엔드 정책

프론트엔드는 역할별 화면을 분리하되, 공통 설정을 받아 학원별로 다르게 렌더링한다.

주요 화면:

- 관리자 세팅 웹
- 원장 관리 웹
- 강사 수업 화면
- 수강생 모바일 웹/PWA

기술 스택 후보:

- React
- Vite
- TypeScript
- Tailwind CSS
- PWA

프론트엔드는 feature-based 구조를 사용한다.

## MVP 범위

초기 MVP는 전용 앱 출시가 아니라 운영이 되는 웹앱에 집중한다.

포함:

- 그룹 수업과 1:1 수업 모두 지원
- 원장 운영 웹
- 수강생 모바일 앱/PWA
- 강사 모드
- Dionomy 회사 웹
- Dionomy 관리자 세팅 웹
- 우리 측 관리자 학원 생성
- 학원별 화이트라벨 설정
- 원장 로그인
- 카카오 로그인
- 수강생 등록
- 강사 등록·초대
- 수업 일정 등록
- 반복 일정 등록
- 수강생 배정
- 출석 처리
- 수강권 관리
- 수강권 자동 만료
- 수강권 사용 이력
- 결석 신청과 승인
- 보강 또는 다른 세션 이동
- 클래스노트 작성·열람
- 공지 작성·발송
- 위험 수강생 신호
- 케어 기록
- 수강생 모바일 웹

제외:

- 앱스토어 출시 자동화
- 완전한 노코드 빌더
- 원장용 화면 편집기
- 원장용 워딩 편집기
- 결제 영역
- 수강권 결제
- 원장 구독 결제
- 자동 케어 발송
- 강사 정산 자동화
- 강사용 이탈 신호 열람
- 개인별 클래스노트 코멘트
- 자료 공유 게시판
- 복잡한 통계
- 다중 지점 관리
- 고급 권한 체계

상세 체크리스트는 [MVP 기능 체크리스트](mvp-checklist.md)를 기준으로 한다.

## 핵심 도메인 결정

- 수업 형태는 그룹 수업과 1:1 수업을 모두 지원한다.
- 수강생 자율예약은 MVP에서 제외한다.
- 원장이 수업을 만들고 수강생을 초대 또는 배정한다.
- 출석체크는 강사가 한다.
- 수강권은 기간과 회차 결합형으로 처리한다.
- 수강권은 회차 소진 또는 기간 만료 중 먼저 도달한 조건으로 만료된다.
- 케어 발송은 MVP에서 제외하고, 위험 신호만 보여준 뒤 원장이 수동으로 관리한다.
- 강사 정산은 월급 고정 기준으로 보고 자동 계산은 MVP에서 제외한다.
- 클래스노트는 수업 단위로 공유한다.
- 그룹 수업은 같은 노트를 공유하고 개인 코멘트는 MVP에서 제외한다.
- 결석 신청은 수강생이 사유와 원하는 결과를 선택해 신청하고, 강사가 승인한다.
- 결석 결과는 다른 세션 이동 또는 보강으로 처리한다.
- 게시판은 공지만 제공하고 자료 공유는 클래스노트로 흡수한다.
- 결제 영역은 MVP에서 제외한다.
- 이탈 신호는 원장만 열람하고 강사 열람은 v2로 둔다.

## 구현 우선순위

### 1차

```txt
공통 운영 기능
관리자 세팅 화면
학원 생성
기능 on/off 설정
워딩 변경 설정
원장 관리자 화면
```

### 2차

```txt
수강생 모바일 웹
예약 확인
일정 변경 요청
잔여 회차 확인
보강권 확인
수업 기록 확인
```

### 3차

```txt
학원별 PWA 브랜딩
푸시 알림
WebView 앱 껍데기
앱 이름/아이콘 적용
앱 출시 정보 관리
```

## 핵심 원칙

- 원장은 앱을 만들지 않고 운영만 한다.
- 우리 측 관리자가 학원별 앱 세팅을 전담한다.
- 학원별 차이는 코드가 아니라 설정값으로 처리한다.
- 공통 웹앱과 공통 백엔드를 우선 만든다.
- WebView 앱은 비즈니스 로직 없는 껍데기로 유지한다.
- MVP는 앱 출시보다 실제 학원 운영 문제 해결에 집중한다.
